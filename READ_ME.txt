How To Use

(NOTE: I prioritized the controller when coding and testing, so as a result the keyboard controls don't feel very good, so please use a controller instead)

Keyboard Controls: 
- Space or N to start the game
- E to accelerate
- S to turn left (I say A in the video, its supposed to be S)
- F to turn right
- W to shoot a Fire Bullet
- R to shoot an Ice Bullet
- N to respawn player ship (only works if player ship is dead)
- B to kill all enemies
- I to spawn a new asteroid
- P to pause
- T to slow rate of time
- O to run a single frame
- F1 to toggle debug rendering
- F8 to reset game
- Esc to return to the attract screen, then Esc again to quit Game

Controller Controls:
- Start or A to start the game
- Left Stick to thrust and turn
- Right Stick to turn, but only when the player isn't thrusting
- Right Trigger to shoot a Fire Bullet
- Left Trigger to shoot an Ice Bullet
- Start to respawn player ship (only works if player ship is dead)
- Select to return to the attract screen, then Select again to quit Game

Deep Learning

My programming style up until coming into Guildhall was to write my code in such a way that it was as elegant and sophisticated as possible. This habit partially came from me getting enjoyment from writing this kind of code, but also because I didn’t have considerations about speed or the fact that others may look at my code. After coming to Guildhall and having KISS beaten into my head again and again, I’ve only actually come to understand the importance of keeping things simple stupid, through my experience coding Starship Gold. I decided I would try to code this assignment with KISS in mind and it ended up being rather effective. Whenever I began working on a new mechanic, I thought to myself, “how should I implement this?” and simply went with whatever was the first thing that came to mind. Precisely because it is my first thought it, more often than not, was indeed a very simple solution that I may have overlooked if I had instead overthought things. Because I was able to code much faster with this methodology, I was able to implement more stuff and have more fun coding said stuff. After using this methodology for this project, I’ve come around to the idea that simple code indeed IS good code. With all that being said, I don't think there's anything inherently wrong with more sophisticated code, and I do still enjoy that kind of programming. So going forward I wanna try to train my brain to come up with simple AND sophisticated solutions, a skill that I’ll hopefully build up naturally as I code more and more and more going forward.
